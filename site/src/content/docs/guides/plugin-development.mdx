Plugins are Rust crates with a `manifest.yaml` that Ralph Engine auto-discovers at startup. Each plugin declares its kind, capabilities, and contributions. The scaffolder generates everything you need — no manual wiring required.

## Scaffold a New Plugin

<Tabs syncKey="pkg">
  <TabItem label="npx">

    npx create-ralph-engine-plugin --name my-tool --publisher acme --kind template

### Flags

| Flag | Required | Description |
|------|----------|-------------|
| `--name` | Yes | Plugin name. Used as crate name and in the plugin ID (`publisher.name`). |
| `--publisher` | Yes | Publisher namespace. Use your org name (e.g., `acme`). |
| `--kind` | Yes | Plugin kind. Determines which capabilities and trait methods are relevant. |

Valid `--kind` values: `template`, `agent_runtime`, `mcp_contribution`, `policy`, `remote_control`.

## Generated Files

The scaffolder creates a complete Rust crate:

```
my-tool/
├── src/
│   ├── lib.rs          # Plugin descriptor + PluginRuntime trait
│   └── i18n/
│       └── mod.rs      # Generated by build.rs (do not edit)
├── locales/
│   ├── en.toml         # English strings
│   └── pt-br.toml      # Brazilian Portuguese strings
├── build.rs            # Compiles locale TOML into Rust code
├── manifest.yaml       # Metadata for auto-discovery
└── Cargo.toml          # Crate dependencies
```

### `src/lib.rs`

The entry point. Declares the plugin descriptor (ID, kind, trust level, capabilities, lifecycle stages) and implements the `PluginRuntime` trait. The scaffolder pre-fills the descriptor constants and wires up the i18n module.

### `manifest.yaml`

Declarative metadata that Ralph Engine reads at startup for auto-discovery. Contains the plugin ID, kind, display name, summary, publisher, trust level, capabilities, and contribution listings. Example:

```yaml
id: acme.my-tool
kind: template
display_name: My Tool
display_name_locales:
  pt-br: Minha Ferramenta
summary: What this plugin does.
summary_locales:
  pt-br: O que este plugin faz.
publisher: acme
trust_level: community
plugin_version: 0.1.0
plugin_api_version: 1.0.0
engine_version: ">=0.2.0-alpha.1"
runtime: true
capabilities:
  - template
```

### `locales/en.toml` and `locales/pt-br.toml`

TOML files with translatable strings. No Rust knowledge needed to add translations. `build.rs` compiles these into type-safe Rust code at build time.

### `build.rs`

Reads every `.toml` file in `locales/` and generates the `i18n` module with compile-time constants. You do not edit this file unless you add new TOML sections.

### `Cargo.toml`

Standard Rust crate manifest. The only required dependency is `re-plugin` (runtime) and `re-build-utils` (build script):

```toml
[dependencies]
re-plugin = { version = "0.2.0-alpha.1" }

[build-dependencies]
re-build-utils = { version = "0.2.0-alpha.1" }
```

## Document Your Plugin

Add a `README.md` to your plugin directory. It will be rendered on the plugin's page on the Ralph Engine website. Use `## Heading` sections to organize content:

```markdown
# acme.my-tool

Short summary.

## Overview

What your plugin does and why someone would use it.

## How to use

Step-by-step instructions.

## Configuration

Available options and examples.
```

For translations, add locale-specific files alongside the default:

- `README.md` — default (English)
- `README.pt-br.md` — Portuguese (Brazil)

If a translated README is not available, the site falls back to the default. The `description` and `description_locales` fields in `manifest.yaml` provide the short description shown in the plugin catalog.

:::note
README content from community plugins is sanitized at build time — HTML tags and scripts are stripped. Only plain text is rendered.
:::

## Implement the Runtime

Every plugin implements the `PluginRuntime` trait. The trait has three methods:

```rust
pub trait PluginRuntime: Send + Sync {
    /// Returns the plugin identifier this runtime belongs to.
    fn plugin_id(&self) -> &str;

    /// Executes a prepare or doctor check.
    fn run_check(
        &self,
        check_id: &str,
        kind: PluginCheckKind,
        project_root: &Path,
    ) -> Result<CheckExecutionResult, PluginRuntimeError>;

    /// Bootstraps an agent runtime session.
    fn bootstrap_agent(
        &self,
        agent_id: &str,
    ) -> Result<AgentBootstrapResult, PluginRuntimeError>;

    /// Registers an MCP server and validates it can start.
    fn register_mcp_server(
        &self,
        server_id: &str,
    ) -> Result<McpRegistrationResult, PluginRuntimeError>;
}
```

Not all methods need real implementations. A template plugin only cares about `run_check` (to validate project files). An agent runtime plugin focuses on `bootstrap_agent`. Return `Err(PluginRuntimeError::new(...))` for operations your plugin does not support:

```rust
fn bootstrap_agent(
    &self,
    agent_id: &str,
) -> Result<AgentBootstrapResult, PluginRuntimeError> {
    Err(PluginRuntimeError::new(
        "not_an_agent_plugin",
        format!("my-tool does not provide agent '{agent_id}'"),
    ))
}
```

## Add Translations

Translations live in `locales/*.toml`. Each file is one language. The TOML structure mirrors the plugin's contributions:

```toml
# locales/en.toml
[plugin]
name = "My Tool"
summary = "What this plugin does."

[template]
name = "Starter"
summary = "Basic project template."
```

```toml
# locales/pt-br.toml
[plugin]
name = "Minha Ferramenta"
summary = "O que este plugin faz."

[template]
name = "Starter"
summary = "Template básico de projeto."
```

Adding a new language means creating one TOML file (e.g., `locales/es.toml`). `build.rs` picks it up automatically — no Rust changes needed.

## Test Your Plugin

Run the plugin's unit tests:

```bash
cargo test -p my-tool
```

Run tests sequentially (useful for tests with shared state):

```bash
cargo test -p my-tool -- --test-threads=1
```

Verify the plugin satisfies the contract tests:

```bash
npm run contracts:verify
```

## Validate the Manifest

Show the resolved plugin metadata:

```bash
ralph-engine plugins show acme.my-tool
```

List all registered plugins (official + community):

```bash
ralph-engine plugins list
```

## Plugin Kinds

| Kind | Contributes | Use when |
|------|------------|----------|
| `template` | Templates, prompts | Project scaffolding and starter files |
| `agent_runtime` | Agents, MCP servers | Agent integration (Claude, Codex, custom) |
| `mcp_contribution` | MCP servers | External tool integration without agent runtime |
| `policy` | Policies, checks | Enforcement rules (TDD, lint, compliance) |
| `remote_control` | Providers | Remote execution (SSH, containers, cloud) |

The `--kind` flag determines which capabilities and runtime hooks the scaffolder pre-configures. You can add more capabilities later by editing `manifest.yaml` and implementing the corresponding trait methods.

## Trust Levels

| Level | Meaning |
|-------|---------|
| `official` | First-party plugins shipped with Ralph Engine. Live in `plugins/official/`. |
| `community` | Third-party plugins. Auto-discovered via `manifest.yaml` in the project. |

Trust level affects default permission grants. Official plugins have full access. Community plugins operate with reduced permissions by default.

## Publish

Community plugins are distributed as standalone Rust crates. To make your plugin available:

1. Publish the crate to [crates.io](https://crates.io) or a private registry.
2. Users add it as a dependency in their project.
3. Ralph Engine discovers it via `manifest.yaml` at startup.

No plugin registry or marketplace is needed. The manifest is the discovery mechanism.
