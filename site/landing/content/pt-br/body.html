Runtime open source orientado a plugins

O runtime para fluxos de desenvolvimento com agentes.

Ralph Engine entrega uma base em Rust para loops de agente, contratos de plugin, integrações com MCP e verificações de qualidade que continuam coerentes do desenvolvimento local até a CI.

Por que isso existe

Ralph Engine está sendo construído para reduzir fragilidade de workflow, não para adicionar mais uma camada dela.

A maior parte dos produtos de IA otimiza a demo primeiro. Ralph Engine toma a direção oposta: contrato de runtime, fronteiras explícitas de plugin, metadados públicos, disciplina de release e docs que acompanham o código.

Legibilidade sob pressão

O caminho sério precisa parecer o caminho padrão. Isso pede CLI limpa, capacidades explícitas, descoberta honesta de plugins e gates de release visíveis do desenvolvimento local até a CI.

O produto público precisa parecer calmo, durável e aberto. As superfícies devem explicar o produto rápido, continuar acessíveis no mobile e dar um caminho coerente para pessoas e agentes.

O que o runtime está otimizando

Contratos de plugin, disciplina de contexto e qualidade de release fazem parte do produto.

A meta não é abraçar toda moda nova de modelo. A meta é dar aos times uma base estável para rodar fluxos com agentes sem abrir mão de clareza, portabilidade e disciplina séria de release.

Core em Rust

O runtime e os plugins oficiais estão sendo reconstruídos em Rust para manter o contrato público forte, testável e explícito.

Ecossistema plugin-first

Plugins oficiais estabelecem a base de qualidade enquanto contratos de terceiros continuam agnósticos de linguagem.

MCP como parte do core

MCP externo, superfícies agregadas e disciplina de contexto pertencem ao runtime, não a um monte de convenção improvisada de prompt.

Gates rígidos de qualidade

Coverage, rustdoc, scanners, docs build, hardening de CI e fluxo de release são tratados como parte do produto.
