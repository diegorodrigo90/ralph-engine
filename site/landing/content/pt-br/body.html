Runtime open source orientado a plugins

O runtime para fluxos de desenvolvimento com agentes.

Ralph Engine entrega uma base em Rust para loops de agente, plugins oficiais, integrações com MCP e verificações de qualidade que continuam coerentes do desenvolvimento local até a CI.

Instalação via código-fonte

git clone https://github.com/diegorodrigo90/ralph-engine.git
cd ralph-engine
./scripts/bootstrap-dev.sh
cargo run -p re-cli -- --version

npm e Homebrew continuam canais oficiais e estão sendo reconstruídos no novo pipeline de release.

Por que isso existe

Ralph Engine está sendo construído para reduzir fragilidade de workflow, não para adicionar mais uma camada dela.

A maior parte dos produtos de IA otimiza a demo primeiro. Ralph Engine está indo na direção oposta: contrato de runtime, fronteiras de plugin, disciplina de release e superfícies públicas que continuam legíveis para pessoas e agentes.

O produto precisa continuar legível quando a pressão aumenta.

Isso significa CLI limpa, capacidades explícitas, superfícies legíveis por máquina e documentação pública que acompanha o código em vez de ficar para trás.

A meta não é abraçar toda moda nova de modelo. A meta é dar aos times uma base estável para rodar fluxos com agentes sem abrir mão de clareza, portabilidade e qualidade de release.

Open source, plugin-first, runtime-first. Essa ordem importa porque adoção depende mais de confiança e coerência do que de espetáculo.

Manifesto do runtime

O caminho sério precisa parecer o caminho padrão.

O ponto não é um shell esperto em volta de chamadas de modelo. O ponto é um runtime que mantém código, plugins, docs, metadados públicos e automação de release sob um contrato explícito.

O que fica estável enquanto a implementação evolui

  • Arquitetura plugin-first. Plugins oficiais evoluem em Rust, contratos de terceiros seguem agnósticos de linguagem e as capacidades continuam explícitas.
  • Qualidade e segurança são partes do produto. SemVer, Actions pinadas, varredura de supply chain, coverage, docs build e automação de release fazem parte do contrato do runtime.
  • O contexto precisa continuar coerente. Docs, metadados de plugins, superfícies do repositório e endpoints legíveis por máquina são alinhados de propósito para que pessoas e agentes encontrem o caminho certo com rapidez.

Como a adoção deve parecer

O produto público precisa parecer calmo, sério e durável. Isso pede entrega estática por padrão, sinais honestos de maturidade e nada de densidade artificial de ecossistema.

Comece pelas docs. Vá para a superfície de plugins quando precisar avaliar extensibilidade. Mantenha GitHub e os endpoints legíveis por máquina por perto quando quiser confirmar os detalhes.
